## üîê Row Level Security (RLS) Summary

All tables have RLS enabled with comprehensive policies:

### User Access Patterns:

**Public (Unauthenticated):**
- ‚úÖ Read: builders, properties (active), reviews (approved)
- ‚úÖ Write: leads, interactions, page_views, events (analytics)
- ‚ùå No access to: user profiles, subscriptions, payments

**Regular Users (Authenticated):**
- ‚úÖ Read: Own profile, own subscriptions, own payments
- ‚úÖ Write: Reviews, events, interactions
- ‚ùå No admin functions

**Builders (role = 'builder'):**
- ‚úÖ Read: Own leads, own property analytics, own properties
- ‚úÖ Write: Own properties, own lead status updates
- ‚úÖ Manage: Own subscription, own profile

**Admins (role = 'admin'):**
- ‚úÖ Full access to: All tables
- ‚úÖ Can: View all leads, analytics, payments
- ‚úÖ Manage: Builders, properties, reviews, users

## üêõ Common Issues & Solutions

### Issue: Migration 006 fails
**Error:** `relation "public.leads" does not exist`  
**Solution:** Apply migration 007 first, then retry 006

### Issue: Column doesn't exist
**Error:** `column "images" does not exist` (or similar)  
**Solution:** Apply migrations 009-011 to extend tables

### Issue: Permission denied
**Error:** RLS policy violation  
**Solution:** 
- Check user role in profiles table
- Use service role key for admin operations
- Verify RLS policies are applied

### Issue: Phone auth not working
**Error:** SMS not sent or invalid OTP  
**Solution:** See PHONE_AUTH_SETUP.md
- Enable Phone provider
- Use Fake OTP for testing
- Check Twilio credentials for production
